# Releasing ScrollSense

How to cut a new release when there are new changes to ship. There are two
artifacts — the **menu-bar app** (`.dmg`, via GitHub Releases) and the **CLI**
(Homebrew tap) — and they can be released together or independently.

## Prerequisites (once per machine)

```bash
./setup-signing.sh   # creates the stable "ScrollSense Self-Signed" cert
gh auth login        # so release.sh can publish to GitHub
```

> Always release from the **same machine / signing cert**. The Accessibility
> grant is keyed to the app's code signature, so signing every release with the
> same cert means users grant permission **once** and it survives updates. A
> different cert resets the grant for everyone on their next update.

## Where versions live

| Spot | Set by |
|------|--------|
| CLI `version:` in `Sources/ScrollSense/ScrollSense.swift` | `./bump-version.sh` (manual source bump) |
| Menu-bar app `Info.plist` | stamped at build time from the release argument (automatic) |
| Homebrew formula (`Formula/scrollsense.rb`) — url + sha256 + assert | `./scripts/release-homebrew.sh` (needs the published tarball's checksum) |

## The flow

1. **Land your changes** (merge/commit the new features or fixes).

2. **Bump the version** in source:
   ```bash
   ./bump-version.sh 1.2.0
   git commit -am "Release v1.2.0"
   ```

3. **Release the app** (builds + signs the DMG, publishes a GitHub release):
   ```bash
   ./release.sh 1.2.0
   ```

4. **Release the CLI** to Homebrew (tags, updates the formula's url + sha256 + test):
   ```bash
   ./scripts/release-homebrew.sh 1.2.0 --tap-dir ../homebrew-scrollsense
   ```

Either release script may run first — both reuse an existing `v1.2.0` tag, so
they won't collide.

## Verify before announcing

```bash
open build/ScrollSense-1.2.0.dmg          # install it, confirm the menu bar works
/Applications/ScrollSense.app             # grant Accessibility, switch devices
brew upgrade scrollsense && scrollSense --version   # if you published the CLI
```

## Writing the changelog

GitHub release notes are generated by `release.sh` (install steps). For
human-facing "what's new," add a short bullet list to the release on GitHub, or
keep a `CHANGELOG.md` and paste the latest section into the release body.

## Build stages individually (no publishing)

```bash
./build-app.sh                # build/ScrollSense.app only
./build-dmg.sh 1.2.0          # build/ScrollSense-1.2.0.dmg only (no GitHub release)
```
